# AI家庭教师 - 技术演进路线

最终目标：上架的移动端 App（iOS/Android）

分 4 个阶段演进，每个阶段独立验证一类风险，避免一次性投入过大。

---

## 阶段 1：HTML 原型（1-3 天）

**目标**：验证产品形态和交互闭环

- 技术栈：纯 HTML + JS + Tailwind CSS
- 运行方式：手机浏览器直接预览，零安装
- 范围：4 个核心页面 + 假数据 + 模拟 AI 对话脚本
  - 学生切换 + 科目首页
  - 拍照/输入提问 + 启发式答疑
  - 错题本 + 知识图谱
  - 复习任务 + 变式题
- **产出**：可点可滑的"伪 App"，拿给目标用户（学生/家长）看反馈
- **验证问题**：
  - 苏格拉底式答疑学生买不买账？
  - 错题本 + 复习的闭环是否真的有人用？
  - 家长监管端是否有价值？

**不做**：真实 AI、真实存储、登录注册

---

## 阶段 2：H5 + 真实 AI（1-2 周）

**目标**：验证 AI 效果和 Prompt 质量

- 在阶段 1 基础上接入大模型 API（推荐 Claude Sonnet 4.6 或 GPT-4o）
- 加一个轻后端（Node/Python + SQLite）存错题和用户数据
- 加 OCR（拍照识题，可用通义 / 百度 OCR）
- 实现核心 Prompt：
  - 启发式答疑 System Prompt
  - 错题知识点提取 Prompt
  - 变式题生成 Prompt
- **验证问题**：
  - 启发式 Prompt 调到能用吗？会不会直接给答案？
  - OCR 准确率够吗？数学公式识别如何？
  - 变式题会不会跑偏？

**不做**：跨端打包、推送、原生相机

---

## 阶段 3：跨端 App 壳（2-4 周）

**目标**：变成真正的 App，能上架

- 阶段 3 前置形态：可安装 PWA/H5 内测壳
  - 已补 `manifest.webmanifest`、`sw.js`、`offline.html` 和移动端安装元信息
  - 作用：用 HTTPS/localhost 快速验证“像 App 一样进入”和弱网离线兜底
  - 边界：PWA 不能替代 iOS/Android 上架包, 相机、推送和应用商店支付仍需 RN/Expo 或原生能力
- **技术选型：React Native + Expo**
  - 理由：H5 的 React 代码可复用 70%+，发布流程比 Flutter 简单
  - 当前已新增 `mobile/` Expo SDK 56 骨架和 `npm run mobile:check`, 先复用后端 OTP 登录、bearer session、dashboard、questions/finish、review-tasks、uploads/OCR、devices、mistakes、knowledge-tree、reports 和 billing API
- 复用阶段 2 的后端 API，前端重写为 RN 组件
- 接入原生能力：
  - 相机（拍照识题）
  - 推送通知（复习提醒的关键载体; 后端已具备设备 token、提醒任务和 HTTP 推送 provider; mobile 已能注册 Expo push token）
  - 本地存储（离线缓存错题）
- **产出**：可打包 iOS/Android 安装包，TestFlight / 内测分发

**关键决策点**：如果阶段 2 用户留存差，说明产品形态有问题，**不要硬上 App**

---

## 阶段 4：长期记忆与个性化（持续迭代）

**目标**：构筑产品定位文档里说的"核心壁垒"

- 引入向量数据库（pgvector / Qdrant）存知识图谱
- RAG：AI 答疑时带上该学生的历史错题上下文
- 复习推送策略调优（艾宾浩斯参数 A/B 测试）
- 家长端独立 App 或小程序
- 多模态升级：语音提问、手写公式识别

---

## 关键原则

1. **不要跳过阶段 1**：直接上 RN 会浪费大量时间在 UI 反复改，纸面设计发现不了真实交互问题
2. **阶段 2 → 3 是分水岭**：验证产品有人用，再投入跨端开发
3. **每个阶段都要拿给真人用**：自己看再多遍也不如用户用 5 分钟
