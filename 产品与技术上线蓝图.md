# AI家庭教师 - 产品与技术上线蓝图

目标: 把当前原型推进成可内测、可上架、可运营的 iOS/Android App。本文按产品经理和项目经理视角说明当前边界、系统架构、数据模型、AI 接入方式和后续阶段验收。

## 1. 当前项目状态

当前项目已经不是单纯静态页面原型, 而是包含三层产物:

- `prototype/`: H5/PWA 原型, 用于快速演示和用户访谈。
- `server/`: Fastify + Prisma + PostgreSQL 后端, 承载账号、题目、错题、复习、报告、订阅、通知和运营后台 API。
- `mobile/`: Expo/React Native 移动端骨架, 复用后端 bearer session API, 覆盖登录、首页、提问、复习、错题、报告、Plus 和我的页。

当前已具备的产品闭环:

- 手机号验证码登录和 session。
- 单学生档案和账号级数据隔离。
- 文本/图片题目创建、OCR 入口和启发式答疑。
- 错题入库、D1/D3/D7/D15 复习任务、变式题答题。
- 首页统计、错题本、知识图谱、家长周报。
- Plus 套餐、订单/订阅雏形、账号导出和注销。
- 运营后台指标、内容审核队列、AI 日志和扩量健康检查。

当前未完成或未通过的上线关键点:

- 本机 Expo/Metro 真实启动验收未通过: `npm run start:check` 超时, Metro 未监听端口。
- 本地数据库烟测未跑通: 当前机器缺少 PostgreSQL/Docker, CI 已配置 PostgreSQL service。
- 真实 AI Key 未配置: 当前默认可走 mock, 需要配置 `LLM_PROVIDER`、`LLM_API_KEY`、`LLM_MODEL` 后执行 `npm run ai:check`。
- OCR、支付、推送、对象存储仍是可替换 provider 或 dev/mock 模式, 内测前需要接入真实服务。

## 2. 前后端分离架构

### 前端层

移动端 App 使用 Expo/React Native。前端只负责交互、相机/相册、推送 token、本地 session 和少量本地缓存, 不直接访问数据库或第三方 AI Key。

前端通过后端 API 访问所有业务能力:

- 认证: `/auth/otp/request`, `/auth/otp/login`, `/auth/logout`
- 当前账号: `/me`, `/me/profile`, `/account/export`, `/account`
- 学习: `/questions`, `/questions/:id/answer/next`, `/questions/:id/finish`
- 复习: `/review-tasks/today`, `/review-tasks/:id/answer`
- 结果: `/dashboard`, `/mistakes`, `/knowledge-tree`, `/reports/parent-weekly`
- 原生能力回传: `/uploads/images`, `/ocr/extract`, `/devices`
- 商业化: `/plans`, `/billing/checkout`, `/billing/status`, `/billing/cancel`

### 后端层

后端是 App 的唯一业务可信边界:

- 校验 session, 解析当前 `userId`。
- 控制所有学习数据必须挂到当前用户。
- 调用 LLM/OCR/支付/推送 provider。
- 写入 AI 调用日志、成本、错误、复习结果和运营指标。
- 对生产环境暴露 `/health` 和 `/ready` 做健康检查。

### 数据层

数据库使用 PostgreSQL, Prisma 管理 schema 和 migrations。学习数据以 `users.id` 为主轴隔离, 通过外键和索引支撑查询、删除、导出和运营统计。

## 3. 数据库表分组

账号与合规:

- `User`: 手机号、昵称、协议版本、同意时间。
- `StudentProfile`: 年级、学段、目标科目、复习提醒设置。
- `Session`: bearer session token、过期和吊销。
- `AuthOtp`: 验证码、尝试次数、过期和消费时间。

学习闭环:

- `Subject`: 科目。
- `KnowledgeNode`: 知识树节点。
- `Question`: 原题、输入类型、图片、OCR 文本、状态。
- `AnswerSession`: 一道题的一次引导过程。
- `AnswerMessage`: 用户和 AI 的结构化消息。
- `ErrorRecord`: 错题、错因、知识点、掌握状态。
- `ReviewTask`: D1/D3/D7/D15 复习任务、变式题、答题结果。

AI 与运营:

- `AiEvent`: provider、model、promptVersion、token、耗时、成本、失败原因。
- `UserFeedback`: 用户反馈、评分、类别和页面。
- `DeviceToken`: App/PWA 推送 token。
- `NotificationDelivery`: 推送投递记录和去重键。

商业化:

- `PaymentOrder`: 订单、金额、支付渠道、支付状态和原始回调。
- `Subscription`: 套餐、权益周期、取消续订和来源订单。

当前表结构已经覆盖内测 MVP 所需的主数据。后续成熟 App 可增加:

- `ParentProfile`: 家长身份、联系方式、与学生关系。
- `LearningGoal`: 阶段目标、考试目标、薄弱点目标。
- `VectorMemory`: 学生历史错题向量和 RAG 索引。
- `ContentModerationCase`: 人工审核工单和处置结果。

## 4. AI 能力接入

AI Key 只放在后端环境变量, 不进入移动端、不进入 Git。

核心配置:

- `LLM_PROVIDER`: `mock`、`openai` 或兼容 OpenAI 协议的 provider 名称。
- `LLM_API_KEY`: 后端调用大模型的密钥。
- `LLM_BASE_URL`: 兼容 OpenAI Chat Completions 的服务地址, 默认 `https://api.openai.com/v1`。
- `LLM_MODEL`: 模型名。
- `LLM_TIMEOUT_MS`: 请求超时。
- `LLM_MAX_OUTPUT_TOKENS`: 单次最大输出。
- `LLM_READY`: 真实 AI 上线门禁, 只有 `ai:check`、`eval:ai` 和人工抽检通过后才能设为 `true`。

后端已有三类 Prompt 能力:

- 启发式答疑: `generateSocraticStep`
- 错因提取: `extractErrorRecord`
- 变式题生成: `generateVariantQuestion`

验收命令:

- `cd server && npm run ai:check`: 验证真实 provider 是否可用。
- `cd server && npm run eval:ai -- --output evals/reports/latest.json`: 评测泄题率、引导率、错因具体率、变式一致率和内容安全。

产品验收标准:

- 不直接泄露最终答案, 先引导学生分析条件和知识点。
- 错因不能只写“粗心”或“不会”, 必须可指导复习。
- 变式题必须考察同一知识点, 且只有一个正确答案。
- 每次真实 AI 调用必须记录 `AiEvent`, 便于成本、失败率和质量追踪。

## 5. 阶段性上线计划

### 阶段 A: 架构基线冻结

目标: 明确前后端边界、数据库主表、AI provider 接入方式。

验收:

- `产品与技术上线蓝图.md` 与 `阶段2技术设计.md` 对齐。
- `server/prisma/schema.prisma` 覆盖账号、学习、AI、通知和商业化主表。
- `server/src/app.js` 注册核心业务 API。
- `mobile/src/api/client.js` 只通过后端 API 访问业务数据。

### 阶段 B: 本地可运行闭环

目标: 开发机可启动后端、数据库和移动端, 进入演示模式或真实 API 模式。

验收:

- PostgreSQL 可用后执行 `cd server && npm run verify:db`。
- Expo 启动问题解决后执行 `cd mobile && npm run start:check` 通过。
- 移动端能完成登录、提问、加入错题、今日复习、查看报告。

### 阶段 C: 真实 AI 内测准备

目标: 替换 mock AI, 用真实题目评测启发式答疑质量。

验收:

- 配置真实 `LLM_API_KEY` 后 `npm run ai:check` 通过。
- `eval:ai` 最新报告通过。
- 真实题目人工抽检至少 50 道, 记录在 `AI评测记录模板.md`。

### 阶段 D: 小规模内测

目标: 支持 20-100 名学生/家长连续使用。

验收:

- 关闭 mock 登录和 legacy user id。
- 配置真实短信/验证码、对象存储、推送和支付 provider。
- `/admin/metrics`、`/admin/ops-health` 能追踪留存、复习完成率、AI 失败率、成本和反馈评分。

### 阶段 E: 上架准备

目标: 形成可提交应用商店的 App 包和合规材料。

验收:

- TestFlight/Android 内测包可安装。
- 隐私政策、用户协议、未成年人说明和权限说明与真实 SDK 一致。
- 支付、推送、账号注销、数据导出、内容安全完成上架前核对。

## 6. 项目管理汇报节奏

后续每完成一个阶段性任务, 按以下格式汇报:

- 本阶段目标。
- 已完成内容。
- 验证命令和结果。
- 当前风险或阻塞。
- 下一阶段建议。

当前下一优先级:

1. 解决 Expo/Metro 启动链路, 让移动端可以真实打开。
2. 启动 PostgreSQL, 跑通 `verify:db`。
3. 配置真实 LLM Key, 跑通 `ai:check` 和 `eval:ai`。
4. 再进入 UI 打磨和小规模用户内测。
