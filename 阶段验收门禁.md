目标: 用可执行命令判断项目是否能进入下一阶段, 避免只看页面或文档就误判为可内测、可上架。

## 阶段推进规则

### 进入本地闭环验证前

必须满足:

- `cd server && npm run runtime:check` 通过。
- `docker compose up -d postgres` 可启动 PostgreSQL, 或 `.env` 中 `DATABASE_URL` 指向可用 PostgreSQL。

### 进入真实 AI 内测前

必须满足:

- `cd server && npm run verify:db` 通过。
- `cd mobile && npm run start:check` 通过。
- `cd server && npm run ai:check` 通过且 provider 不是 `mock`。
- 真实模型 `eval:ai` 通过, 并完成 50 道真实题人工抽检。
- `LLM_READY=true`, 且 `cd server && npm run deploy:check -- --profile internal` 通过。

### 进入 20-100 人小规模内测前

必须满足:

- `ALLOW_MOCK_LOGIN=false`。
- `ALLOW_LEGACY_USER_ID_AUTH=false`。
- `INTERNAL_TEST_INVITE_CODE` 已配置。
- `CORS_ALLOWED_ORIGINS` 收口到真实前端和管理后台来源。
- OCR、推送、对象存储至少有内测可用 provider。
- `ADMIN_TOKEN` 不是默认值。
- `/admin/ops-health` 可用于扩量前检查。

### 进入应用商店上架前

必须满足:

- TestFlight/Android 内测包可安装并完成核心流程。
- 支付 provider、退款和对账完成真实联调。
- 隐私政策、用户协议、未成年人说明和权限说明与实际 SDK/权限一致。
- 账号注销、数据导出、上传图片清理和日志保留策略验证通过。

## 汇报格式

每完成一个门禁, 汇报:

- 本次处理的门禁。
- 命令和结果。
- 是否允许进入下一阶段。
- 仍然阻塞的事项。

也可以执行 `cd server && npm run stage:status` 生成当前阶段门禁汇总。输出中的 `ready` 表示工程门禁已就绪, `needs-config` 表示需要内测/生产环境变量, `external-blocked` 表示需要本机或第三方服务, `missing` 表示仓库内缺少必要资产。
